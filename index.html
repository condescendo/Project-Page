Object-Centric
Inverse Rendering in Gaussian Splatting

An ongoing project under the chair of Augmented Vision @ DFKI and RPTU kaiserslautern

Core Objective

This project engineers a low-level extension to the Global Illumination Gaussian Splatting (GI-GS) pipeline. By mathematically decomposing scenes into discrete "Object" and "Background" entities at the rasterization level, we move beyond simple view synthesis to full inverse rendering.

Standard Pipeline

Traditional 3DGS bakes lighting and geometry into a single representation. The scene is static; objects cannot be moved, and lighting cannot be changed without significant artifacts.

This Extension

Allows for Asset Extraction without background artifacts, Scene Compositing into novel environments, and Physically-Based Relighting by solving for material properties (Albedo, Roughness, Metallic).

Systems Engineering & CUDA Architecture

The primary contribution is not just calling PyTorch functions, but modifying the underlying C++/CUDA rasterization backend to support dynamic visibility tracking and memory management.

1. Custom CUDA Kernels

To achieve object isolation without expensive post-processing, I rewrote the core renderCUDA kernel.

  • Device Memory Management: Allocated a custom boolean tensor (seen) directly on the GPU to track per-Gaussian visibility persistence across the training lifecycle.
  • Zero-Copy Logic: Visibility logic remains entirely on the GPU during the forward pass, syncing to the host (Python) only when topological changes (pruning) are required.

2. Implementation Snapshot: Thread-Level Visibility

By injecting logic into the forward rasterization pass, the kernel now performs atomic checks. If a thread contributes to a pixel (alpha > threshold), it writes directly to the global memory buffer.

// Inside __global__ void renderCUDA(...)

// 1. Calculate pixel and thread identifiers
auto block = cg::this_thread_block();
uint32_t pix_id = W * pix.y + pix.x;

// 2. Iterate over Gaussian batches (Collaborative fetching)
for (int i = 0; i < rounds; i++, toDo -= BLOCK_SIZE) {
    __syncthreads(); // Sync shared memory load

    // 3. Per-thread processing
    for (int j = 0; !done && j < min(BLOCK_SIZE, toDo); j++) {
        
        // Compute alpha contribution (Equation 2 from 3DGS paper)
        float alpha = min(0.99f, con_o.w * exp(power));
        
        // CUSTOM VISIBILITY LOGIC:
        // Direct buffer access to mark geometry as 'seen' if it contributes to the pixel.
        // Utilizes the rasterizer's inherent depth ordering to avoid raycasting.
        if (alpha > 0.0f) {
            seen[collected_id[j]] = true; 
        }
        // ... (Standard blending continues) ...
    }
}

3. Optimization & Data Flow

Occlusion-Aware Pruning: A Python-side controller aggregates the CUDA-generated seen tensor via C++ bindings. Points that fail to register a true status in the GPU buffer are pruned, reducing VRAM usage and eliminating internal geometry artifacts.

Masked PBR Loss: I engineered a custom loss pipeline that applies segmentation masks before gradient calculation. This stabilizes the non-convex optimization of PBR parameters (Albedo, Roughness) by preventing the solver from fitting noise in empty space.

Key Results

/ Asset Extraction
Clean Geometry

Clean separation of geometry without background artifacts or "floaters," enabling direct export to standard 3D engines.

/ Result
PBR Relighting

Extracted objects react realistically to new lighting conditions because the pipeline solves for intrinsic material properties (Albedo, Roughness).

/ Optimization
Occlusion Pruning

Automatic detection and deletion of internal geometry hidden inside the object, resulting in lighter, hollow meshes.

/ Pipeline
Pybind11 Integration

Seamless bridge allowing Python to control high-level topology logic while C++ handles the heavy pixel-wise atomic operations.
